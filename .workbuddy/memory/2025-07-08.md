- 整合 gzh-design 技能的 6 个主题（摸鱼绿/红白色系/石墨极简风/留白禅意风/摸鱼票据风/橄榄手记）

- 浏览器端合规校验（JS 移植 validate_gzh_html.py）

- 流式 LLM 响应（SSE）

### 技术决策
- 选 React+Vite+TS：组件化、类型安全、TipTap 生态
- 富文本用 TipTap（ProseMirror）：粘贴处理好、可扩展
- HTML→Markdown 用 Turndown.js：纯本地无需 LLM
- 模型供应商用接口抽象：方便后期扩展
- 主题文件放 public/themes/，运行时 fetch 加载并缓存
- skill-core.md 作为 LLM 排版核心指令

### 项目结构
- src/types/ - 核心类型定义
- src/store/ - Zustand 状态管理（theme/editor/settings）
- src/lib/llm/ - LLM 客户端+供应商+提示词构建
- src/lib/storage/ - IndexedDB+加密+自定义主题
- src/lib/validation/ - HTML 合规校验
- src/lib/clipboard/ - 剪贴板+下载
- src/lib/themes/ - 内置主题数据
- src/components/layout/ - 四面板布局
- src/components/topbar/ - API弹窗+自定义主题弹窗
- src/components/editor/ - TipTap+Markdown编辑器
- public/themes/ - 6个主题组件库+通用组件+核心指令

### 修复
- 修复 LLM 排版结果提取逻辑：原正则只取第一个 `<section>`，导致多 section 文章被截断。改为提取所有 `<section>` 块并拼接。
- 在排版提示词中明确要求模型输出完整文章、不得省略章节。

### 修复渲染错乱
- 问题原因：用惰性正则拼接多个 `<section>` 块，破坏了嵌套 section 结构（外层 section 被提前闭合）。
- 改用浏览器 `DOMParser` 提取 `body.innerHTML`，完整保留 section 嵌套关系与多个 section。
- 同时将 `max_tokens` 从固定 8192 改为按模型配置读取，避免长文章被截断。

### 修复长文章截断与警告展示
- 增强排版提示词：要求严格按 Markdown 章节顺序逐段渲染，不得跳过任何内容
- LLM 客户端解析 SSE 中的 `finish_reason`，当因 `length` 截断时给出明确提示
- 右侧面板校验 footer 增加警告详情展开，可查看具体警告内容

### 修复渲染与微信兼容 - 回归原始 skill 流程
- 移除 wechatSanitizer（错误地移除了 `display:flex` 等 CSS），用户确认原始 skill 渲染的 HTML 在微信正常
- 重构 `extractHtml`：不再用 DOMParser（可能修改原始 HTML），改用 `indexOf/lastIndexOf` 截取第一个 `<section` 到最后一个 `</section>` 之间的原始内容，完美保留嵌套结构且不修改任何 HTML
- 确保最终复制的 HTML 与 LLM 原始输出一致，不做额外处理
- 根因：V4 默认开启 thinking 模式，排版任务不需要推理链，导致首 token 延迟极高（几十秒甚至超时）
- 修复：对 `deepseek-v4-flash` / `deepseek-v4-pro` 请求自动添加 `thinking: {type: 'disabled'}`
- 新增 `fetchWithTimeout`：60 秒网络超时，避免请求永挂
- 超时或 AbortError 会正确展示错误信息而非卡死在「发送请求」状态
